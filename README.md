# 🛡️ Aegis Vault (宙斯盾保险库)

[![License](https://img.shields.io/badge/license-Apache--2.0-blue.svg)](LICENSE)
[![Python](https://img.shields.io/badge/python-3.9+-blue.svg)](https://www.python.org/)
[![Docker](https://img.shields.io/badge/docker-supported-green.svg)](https://www.docker.com/)

**Aegis Vault** 是一个面向高隐私场景的自托管加密保险库。它提供文件级混合加密、签到倒计时、胁迫销毁和物理粉碎能力，目标是在“系统被夺取控制权”或“使用者失联”的场景下，优先确保敏感数据不落入他人手中。

当前版本是一个小型单体 FastAPI 应用，强调部署简单、逻辑明确和安全边界可追踪。

---

## 📖 项目定位

传统加密存储通常默认一个前提: 只要口令不泄露，数据就是安全的。但在强制交出口令、设备被扣押、使用者长期失联等场景下，单纯的“加密后静置”并不够。

Aegis Vault 试图补上这层防线:

- **Dead Man's Switch**: 在规定时间内未完成签到，系统触发销毁。
- **Duress Destruction**: 输入预设胁迫码时，系统立即销毁密钥和密文。
- **Self-hosted**: 数据、密钥和状态文件都由你自己的主机保存。

这不是一个通用云盘，也不是成熟的企业级零信任文件平台。它更接近一个“极简高压场景保险库”。

---

## ✨ 当前功能

- 管理员首次初始化系统，创建管理员密钥和签到协议。
- 固定多用户模型，支持 `admin`、`user1`、`user2`、`user3` 四个账号槽位。
- 每个用户拥有独立 RSA 密钥对和独立保险库目录。
- 文件上传时使用 AES-256-GCM 加密，并用用户公钥封装随机 AES 密钥。
- 登录成功后使用服务端会话访问，不再把明文密码传回前端。
- 支持签到重置计时器。
- 支持设置和更新胁迫销毁码。
- 支持物理粉碎密钥和密文文件。
- 支持移动端和桌面端的自适应界面。

## 🔐 安全模型

### 1. 加密模型

- 每个用户拥有一对 **RSA-4096** 密钥。
- 每个上传文件都会生成一个新的 **AES-256** 随机密钥。
- 文件内容通过 **AES-GCM** 加密，获得机密性和完整性校验。
- AES 密钥通过该用户的 RSA 公钥加密后与密文一起存储。
- RSA 私钥使用用户密码加密后保存在磁盘。

这意味着:

- 不同用户之间的文件默认不能互相解密。
- 同一用户的不同文件也不会复用同一个文件密钥。
- 即便拿到加密文件，没有对应私钥和密码也无法正常解密。

### 2. 会话模型

当前版本已从“前端隐藏字段传密码”改为“服务端会话”:

- 登录成功后，服务端生成随机 `session_id`。
- 浏览器仅保存 `HttpOnly` Cookie，不再拿到用户明文密码。
- 会话内容保存在进程内存中，包含:
  - 当前用户
  - 会话过期时间
  - CSRF Token
  - 当前登录口令的内存副本

注意:

- 这个设计已经比旧版安全得多，但仍然属于**进程内内存会话**。
- 服务重启后所有登录态会失效。
- 当前版本没有引入 Redis 或数据库会话持久化。

### 3. 多用户机制

当前项目不是开放注册模型，而是**固定 4 用户槽位模型**:

- `admin`
- `user1`
- `user2`
- `user3`

机制说明:

- 只有 `admin` 登录后才能管理其他用户密钥。
- 每个用户的密钥保存在 `keys/<username>.key` 和 `keys/<username>.pub`。
- 每个用户的加密文件保存在 `vault/<username>/`。
- 登录时，系统会尝试用输入口令解开支持列表中的用户私钥，匹配成功后确定身份。

这套模型的特点:

- 简单直接，便于自托管和审计。
- 权限边界清晰，适合小规模固定成员场景。
- 不支持无限新增用户，也不支持角色系统或细粒度 ACL。

### 4. Dead Man's Switch

后台线程每 60 秒检查一次系统状态:

- 读取 `status.json` 中的 `last_checkin`
- 比较当前时间与最后签到时间差
- 如果超出 `CHECKIN_TIMEOUT`，执行 `destroy_all()`

只要签到成功，系统会重置计时器。

### 5. Duress Destruction

胁迫销毁码支持两种触发入口:

- 在签到入口输入胁迫码
- 在登录入口输入胁迫码

一旦命中:

- 立即执行销毁逻辑
- 清空会话
- 前端只收到失败结果，不回显真实销毁细节

### 6. 销毁模型

项目优先使用系统 `shred` 工具执行文件覆盖删除:

- 密钥文件执行 **3 轮覆盖**
- 密文文件执行 **1 轮覆盖**
- 覆盖后删除文件，再删除目录

如果系统没有 `shred`:

- 会退化为直接删除文件
- 这意味着“逻辑删除”仍然发生，但“物理粉碎效果”会变弱

## 🛡️ 安全架构详解

### 当前安全加固点

当前版本已经完成以下应用层安全修复:

#### 认证与会话安全
- **去除前端明文密码传递**
  - 前端不再通过隐藏字段回传密码
  - 上传、下载、管理操作均使用服务端会话
  
- **会话绑定客户端特征** ✨ (v1.1.0新增)
  - 会话绑定到客户端 IP 地址 + User-Agent 哈希
  - 客户端特征变更时自动失效，防止会话盗用
  - 异常会话使用记录到审计日志

- **CSRF 防护**
  - 已登录后的表单操作都要求 CSRF Token

#### 数据保护
- **状态文件完整性保护** ✨ (v1.1.0新增)
  - status.json 使用 HMAC-SHA256 签名保护
  - 防止攻击者篡改以禁用自毁机制
  - 篡改检测后立即触发销毁

- **私钥内存保护** ✨ (v1.1.0新增)
  - mlock() 防止私钥交换到磁盘
  - 仅 Linux 系统启用，自动降级
  - 登出时立即清除内存

- **审计日志链式哈希** ✨ (v1.1.0新增)
  - 每条日志包含链式哈希，防篡改
  - 日志任何修改都会破坏链的完整性
  - 支持 verify_audit_chain() 验证

#### 路径与文件安全
- **路径穿越防护**
  - 上传和下载文件名都经过规范化
  - 拒绝包含目录跳转的非法路径

- **销毁命令安全化**
  - 删除了 `os.system("find ... -exec shred ...")` 这种 shell 拼接执行方式
  - 改为逐文件安全调用 `shred`

- **下载明文文件清理**
  - 下载时生成的临时明文文件会在响应完成后删除

#### 访问控制
- **危险接口鉴权**
  - `/reset` 在系统未销毁时必须管理员会话才可调用

- **限流**
  - 登录和签到都增加了基于来源地址的失败限流
  - 5次失败后 15 分钟内禁止重试

- **用户密钥误重置保护**
  - 当目标用户已有加密文件时，管理员不能直接重建该用户密钥，避免造成历史文件永久不可解

#### 通信安全
- **安全响应头**
  - 设置了 `CSP`、`X-Frame-Options`、`X-Content-Type-Options`、`Referrer-Policy` 等
  - 支持 HSTS (HTTP Strict-Transport-Security)
  - 禁用缓存防止敏感信息泄露

### 即使物理控制服务器也难以破局

Aegis Vault 的设计目标是：**即使攻击者获得 root 权限控制整个服务器，也应该尽可能地保护已销毁用户的数据**。

#### 攻击场景分析

| 场景 | 防护措施 | 效果 |
|-----|--------|------|
| **读取加密的私钥文件** | RSA-4096 + PBKDF2 + AES-256-CBC 加密 | 🟢 需要用户密码，暴力破解不可行 |
| **修改 status.json 禁用自毁** | HMAC-SHA256 签名保护 + 自动销毁 | 🟢 篡改即触发销毁 |
| **从内存提取已登录用户的私钥** | mlock() + 会话绑定 + gc.collect() | 🟡 大幅降低风险，但需进程重启完全清除 |
| **盗用会话 Cookie 访问文件** | HttpOnly + 会话绑定 (IP/UA) | 🟢 Cookie 盗用后客户端特征仍需匹配 |
| **修改审计日志隐藏踪迹** | 链式哈希 SHA256 防篡改 | 🟢 任何修改都会破坏链 |
| **破坏密钥生成逻辑** | 固定 RSA-4096 配置，无法"降级" | 🟢 无法强制弱化加密 |
| **旧文件用新密钥解密** | 密钥轮转后旧文件自动锁定 | 🟢 新私钥无法解旧文件 |
| **离线磁盘访问** | 依赖全磁盘加密 (FDE) | ⚠️ 应用层无法防护，需操作系统支持 |

#### 最坏情况下的防线

假设攻击者已获得:
- ✅ root 权限读取所有文件
- ✅ 物理访问权限
- ✅ 磁盘完整镜像
- ✅ 进程内存转储

**Aegis Vault 仍能保护：**

1. **未登录的用户数据** ✅
   - 私钥文件加密，需密码
   - 密文文件需要对应私钥解密
   - 密钥文件本身也是 RSA-4096 加密的

2. **已注销用户的内存** ✅
   - 登出时私钥被清除 + gc.collect()
   - 除非攻击者在登出前立即获取内存转储
   - 时间窗口极小

3. **被销毁系统的所有数据** ✅
   - 密钥文件已被 shred 覆盖 3 轮
   - 密文文件已被 shred 覆盖 1 轮
   - 即使是磁盘镜像也无法恢复

4. **被篡改的系统状态** ✅
   - status.json HMAC 签名被破坏时自动销毁
   - 审计日志链式哈希暴露篡改行为

**攻击者需要的条件：**
- 知道用户密码（暴力破解 12+ 字符密码不可行）
- 在用户登出前获取内存（时间窗口受限）
- 破解 RSA-4096（数学上不可行）
- 修改审计日志链（破坏完整性，轻易暴露）

---

### 安全边界说明

**Aegis Vault 保护的范围：**
- 已加密的文件内容
- 已加密的私钥文件
- 会话的盗用
- 日志的篡改
- 状态的伪造

**Aegis Vault 不保护的范围：**
- 全磁盘加密 (FDE) — 需要操作系统和硬件支持
- 物理 Glitching 攻击 (寒冻 DRAM)
- 侧信道攻击 (Spectre/Meltdown 等)
- 社交工程 (强制用户交出密码/胁迫码)
- 代码漏洞 (未来的 0day)

## 🚀 部署方式

### 方案一: Docker Compose

确保宿主机已安装 Docker 和 Docker Compose。

1. 克隆项目并进入目录
2. 准备配置

```bash
cp .env.example .env
```

3. 启动服务

```bash
docker-compose up -d
```

默认访问地址:

```text
http://localhost:46746
```

### 方案二: 源码部署

建议在 Linux 环境运行，因为项目依赖 `shred`。

1. 安装系统依赖

```bash
sudo apt-get install coreutils findutils
```

2. 创建虚拟环境并安装依赖

```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

3. 启动服务

```bash
python3 main.py
```

## 🔧 配置说明

`.env.example` 当前支持如下配置:

| 变量名 | 默认值 | 说明 |
| :--- | :--- | :--- |
| `PORT` | `46746` | Web 服务监听端口 |
| `AEGIS_DATA_DIR` | `./data` | 密钥、密文和状态文件存储目录 |
| `TEMPLATE_DIR` | `./templates` | HTML 模板目录 |
| `CHECKIN_TIMEOUT` | `72` | 签到超时时间，单位小时 |
| `SESSION_TTL_HOURS` | `12` | 登录会话有效期，单位小时 |
| `SESSION_COOKIE_NAME` | `aegis_session` | 会话 Cookie 名称 |
| `SESSION_COOKIE_SECURE` | `false` | 是否仅通过 HTTPS 发送 Cookie |
| `MAX_VAULT_SIZE_MB` | `1024` | 整个保险库总容量上限 |
| `MAX_UPLOAD_SIZE_MB` | `64` | 单文件上传大小上限 |

生产建议:

- 如果通过反向代理启用 HTTPS，建议把 `SESSION_COOKIE_SECURE=true`
- `AEGIS_DATA_DIR` 应配置到持久化磁盘或挂载卷
- 如果使用 SSD / CoW 文件系统，需要清楚 `shred` 的物理覆盖效果可能有限

## 🧭 使用流程

### 首次启动

1. 打开首页
2. 设置管理员密码
3. 设置签到码
4. 完成初始化
5. 使用管理员密码登录

### 创建多用户

1. 使用 `admin` 登录
2. 在“系统管理协议”中选择目标用户
3. 设置该用户的新访问密码
4. 提交后生成该用户专属密钥

注意:

- 当前版本固定只支持 `admin`、`user1`、`user2`、`user3`
- 如果某个用户已经有加密文件，系统会阻止直接重建该用户密钥

### 上传和下载

1. 登录对应用户
2. 选择文件上传
3. 文件会写入该用户独立保险库目录
4. 下载时服务器会临时解密并返回文件
5. 响应结束后临时明文文件会自动清理

### 更新签到码和胁迫码

仅管理员可操作:

- 更新签到码
- 更新胁迫销毁码
- 重置系统

## ⚠️ 已知限制

- 当前用户模型是**固定槽位**，不是动态注册系统
- 当前登录识别方式仍然是“尝试用口令解开支持列表中的私钥”
- 会话保存在进程内存，不适合多实例水平扩展
- 没有数据库，也没有审计日志系统
- 没有文件版本控制、回收站和分享能力
- 下载过程需要在服务端短暂生成明文临时文件
- `shred` 在 SSD、快照盘、CoW 文件系统上并不保证等价于传统机械盘的可恢复性防护

## 🛡️ 适用场景建议

更适合:

- 少量固定成员的小团队
- 个人自托管敏感文件保险库

- 大规模多租户 SaaS
